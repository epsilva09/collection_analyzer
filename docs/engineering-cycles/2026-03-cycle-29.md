# Engineering Cycle Record — 2026-03 Cycle 29

## Cycle Metadata

- Cycle ID/Name: 2026-03 Cycle 29

- Related issue/task: Progress tracking MVP (daily snapshots history)
- Branch/PR: `main`

## Goals

- Introduce a first MVP for collection-progress tracking over time.
- Persist one snapshot per character/locale/day.
- Display a compact timeline on the progress screen.
- Use `character_idx` as canonical key instead of character name.

## Implemented Changes

### 1) Performance

- Snapshot history lookup uses indexed key by `character_idx` and locale.
- Daily upsert avoids duplicate rows for same character/day.

### 2) Quality / Reliability

- Added service tests for:
  - metrics computation (`total`, `completed`, completion rate)
  - daily upsert behavior
  - history ordering and limit
- Controller integration test now asserts history section rendering in progress page.

### 3) Architecture / Maintainability

- Added model and service for timeline persistence:
  - `CollectionProgressSnapshot`
  - `CollectionProgressTrackingService`
- Added migration to rekey snapshot uniqueness from name to `character_idx`:
  - `20260301123000_rekey_collection_progress_snapshots_by_character_idx.rb`

### 4) UX / Accessibility

- Added history card on progress screen with:
  - date
  - completed collections over total
  - completion percentage
  - delta versus previous snapshot (rate and completed collections)
- Added row-level action link (`Ver alterações`) in progress history to open a dedicated changes screen.
- Added dedicated changes view (`progress_changes`) listing per-collection diffs against the previous snapshot:
  - progress variation
  - progress bucket variation
  - materials needed variation
- Added history filters by day/hour in progress timeline to locate specific change windows.
- Added snapshot change classification (`has_changes`, `changes_count`) by comparing each snapshot with the previous one.
- Timeline now defaults to "changed only" visibility, with an option to display all snapshots.
- Added change-type filter in progress changes screen (`added` / `updated` / `removed`).
- Evolved snapshots from day-only to timestamp-based capture (`captured_at`) enabling multiple snapshots per day.
- Added persistent tracked characters registry (`tracked_characters`) updated on character queries.
- Added recurring collection job (`CollectTrackedCharactersSnapshotsJob`) ready to run in scheduled slots.
- Added locale entries (`pt-BR` / `en`) for new history labels.

## Risks / Trade-offs

- Timeline currently captures one snapshot per day, so intra-day trend details are intentionally out of scope for MVP.
- History is shown only when snapshot data exists for the selected character.
- Scheduler integration depends on the selected Active Job backend (Sidekiq or Solid Queue).
