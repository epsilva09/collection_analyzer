## Goals

- Stop trusting inconsistent `values[]` and `rewards[*].applied` fields from the collection payload.
- Make index, compare, and snapshot flows resolve unlocked rewards from collection `progress`.
- Recompute summary values from normalized collection rewards so displayed totals match the game.
- Record follow-up work for broader payload diagnostics and operator visibility.

## Implemented Changes

### 1) Performance

- Centralized reward-threshold calculation in one resolver so index, compare,
  and snapshot paths do not each recompute their own branching rules.
- Kept fallback behavior cheap by reusing raw `values[]` only when collection
  `data` is unavailable.

### 2) Quality / Reliability

- Added a shared `CollectionRewardResolver` to normalize collection rewards by
  `progress`, with controlled fallback for malformed or incomplete reward data.
- Rebuilt summary values from the normalized rewards to avoid undercounting
  collections such as `Solo Flamejante II` and `Cheque o mec. de atk` at 60%.
- Removed drift between screens by making the controller, compare service, and
  snapshot service use the same reward-resolution rule.
- Added regression tests for divergent payloads where top-level `values[]`
  reported `45%` while reward tiers resolve to `48%`.
- Fixed `CollectionProgressSnapshot.for_hour` to be adapter-aware
  (`EXTRACT(HOUR FROM ...)` for PostgreSQL, `strftime` fallback for SQLite),
  restoring full-suite compatibility after the PostgreSQL migration.

### 3) Architecture / Maintainability

- Introduced `app/services/collection_reward_resolver.rb` as the single source
  of truth for:
  - reward thresholds by reward count
  - unlocked reward determination
  - collection summary value aggregation
- Simplified `CollectionSnapshotService` and `CompareCollectionsService` by
  removing duplicated reward-threshold logic.

### 4) UX / Accessibility

- The main Armories index and compare experience now surface collection totals
  consistent with in-game progress for affected reward attributes.
- Collection comparison bonus rows now inherit the same unlock logic used by
  the index summary, reducing conflicting signals in the UI.

## Risks / Trade-offs

- Collections with unusual reward structures beyond the known 1/2/3-tier
  shapes still use a generic evenly distributed fallback threshold rule.
- If the upstream API later adds an explicit threshold field, this resolver
  should be updated to prefer payload-native thresholds over inferred ones.
